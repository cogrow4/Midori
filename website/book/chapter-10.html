Chapter 10: Error Handling

No Null Pointers

Midori does not have null or undefined. Instead, it uses the Option and Result types to represent the possible absence of a value or the possibility of failure. This eliminates an entire category of runtime crashes — every possible failure is encoded in the type system.

Option

Option<T> represents a value that may or may not exist. It has two variants: Some(value) and None. You must explicitly handle both cases; the compiler will not let you forget.

fn find_user(id: Int) -> Option<User> {
    if id == 0 { None }
    else        { Some(User { name: "Alice", id }) }
}

match find_user(42) {
    Some(user) => println("Found {user.name}"),
    None       => println("User not found"),
}

Result

Result<T, E> represents an operation that can succeed with a value of type T or fail with an error of type E. It has two variants: Ok(value) and Err(error).

fn divide(a: Int, b: Int) -> Result<Int, Str> {
    if b == 0 {
        Err("division by zero")
    } else {
        Ok(a / b)
    }
}

fn main() {
    match divide(10, 2) {
        Ok(val) => println("Result: {val}"),
        Err(msg) => println("Error: {msg}"),
    }
}

Short-circuiting with ?

Manually matching on every Result can be verbose. The ? operator provides a concise way to propagate errors: if the value is Ok, it unwraps; if Err, it returns early from the enclosing function.

fn read_config() -> Result<Config, Error> {
    let file   = read_file("config.toml")?
    let config = parse_toml(file)?
    Ok(config)
}

// Equivalent to:
fn read_config_verbose() -> Result<Config, Error> {
    match read_file("config.toml") {
        Ok(file) => {
            match parse_toml(file) {
                Ok(config) => Ok(config),
                Err(e)     => Err(e),
            }
        }
        Err(e) => Err(e),
    }
}

Combining Option and Result

Option and Result compose naturally. The standard library provides helper methods like ok_or, and_then, and map for chaining operations.

fn lookup_zip(city: Str, state: Str) -> Result<Int, Str> {
    db_find_city(city, state)
        |> ok_or("city not found")
        |> and_then(fn(record) {
            record.zip
                |> ok_or("zip code missing")
        })
}

Custom Error Types

For larger applications, define custom error types using enums. Each variant can carry context about what went wrong.

enum AppError {
    NotFound(Str),
    PermissionDenied,
    Timeout(Int),
    Io(Str),
}

fn process() -> Result<Unit, AppError> {
    let data = fetch_data()?  // may return Err(Io(...))
    save(data)?               // may return Err(PermissionDenied)
    Ok(())
}
